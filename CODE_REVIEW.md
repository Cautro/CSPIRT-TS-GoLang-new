# Code Review — CSPIRT

Дата: 2026-07-14
Объём: `backend/` (Go, Gin, Postgres, Redis) целиком (domain / usecase / adapter / controller),
`frontend/` (React + TS) — обзорно, плюс гигиена репозитория.

Методология: 4 параллельных агента прошли соответственно domain+usecase, adapter+config+main,
HTTP/security, репо-гигиену+frontend. Самые серьёзные находки (5 штук) перепроверены вручную —
чтением исходников напрямую, без доверия к пересказу агента. Один false positive найден и
исключён (см. пометку ниже). Итог: отчёт основан на подтверждённых, а не предполагаемых фактах.

---

## 1. Общая архитектура — что сделано хорошо

Проект последовательно проведён через Clean Architecture, и это лучшая черта кодовой базы:

- Единый паттерн для всех 8 сущностей (`user`, `class`, `note`, `complaint`, `event`, `rating`,
  `schedule`, `auth`/`cache`): `internal/domain/<entity>/repo` (порт-интерфейс) →
  `internal/adapter/postgres/<entity>` (реализация) → `internal/usecase/<entity>` (бизнес-логика,
  зависящая только от интерфейса) → `internal/controller/http/<entity>` (HTTP-обвязка). Это
  выдержано без исключений во всех модулях, включая недавно добавленный Redis-слой.
- Каждый Postgres-запрос параметризован (`$1, $2...`) — ни одного случая конкатенации
  пользовательского ввода в SQL не найдено. Единственное построение имени таблицы через код
  (`scheduleTable()` в `schedulePostgresRepo.go`) идёт через явный whitelist, а не сырой ввод.
- Многошаговые записи там, где до них дошли руки, аккуратно обёрнуты в транзакции:
  `eventsPostgresRepo.go` (`AddEvent`, `EventComplete`, `DeleteEvent` и др.) и
  `schedulePostgresRepo.go` (`RolloverSchedules`, `ResetPlannedSchedules`) используют
  `tx.Begin()/defer tx.Rollback()/tx.Commit()` корректно.
- Пул соединений Postgres сконфигурирован явно (`SetMaxOpenConns(25)`, `SetMaxIdleConns(25)`,
  `SetConnMaxLifetime(5*time.Minute)`), а не оставлен на дефолтах драйвера.
- Пароли хранятся через bcrypt (`DefaultCost`) — без кустарной криптографии.
- JWT-миддлварь явно проверяет `t.Method.(*jwt.SigningMethodHMAC)` перед использованием секрета —
  защита от классической атаки подмены алгоритма (`alg: none` / RS256-confusion).
- `internal/usecase/schedule/usecase.go` — лучший usecase в проекте: `normalizeScheduleFilter`,
  `normalizeScheduleLesson`, `mergeScheduleLesson`, `validateScheduleLesson` — чистые, отдельно
  тестируемые функции с честной валидацией каждого поля. Это тот стандарт, до которого не
  дотягивают остальные сущности.
- `internal/controller/http/schedule/scheduleHandlers.go` — лучший файл-хендлер: общие хелперы
  `optionalIntQuery`, `scheduleTypeFromQuery`, `authorizeScheduleRead` устраняют дублирование,
  которое есть почти везде в других файлах.
- Redis-слой (добавлен в этой же сессии): порт/адаптер разделены по тому же паттерну, что и
  Postgres; система спроектирована на выживание без Redis (`cache == nil` не роняет ни логин, ни
  авторизацию); токены в blacklist хранятся хэшированными (SHA-256), а не в открытом виде.

## 2. Frontend — что сделано хорошо

- Токены хранятся в httpOnly cookie (`credentials: "include"`), а не в `localStorage` — верный
  выбор против XSS-кражи токена; `localStorage`/`sessionStorage` для авторизации не используется
  нигде (проверено grep'ом по всему `frontend/src`).
- `dangerouslySetInnerHTML` не используется нигде — не найдено очевидных XSS-инъекций в
  просмотренных файлах.
- Валидация запросов/ответов через `zod`-схемы (`features/auth/api/auth_api.ts` и др.) —
  сервер не считается доверенным источником формы данных вслепую.
- `eslint-plugin-security` и `eslint-plugin-no-unsanitized` подключены в `package.json` —
  осознанный выбор линтеров под безопасность.
- Директории `core/ features/ pages/ shared/` — последовательная feature-based архитектура,
  каждая фича самодостаточна (`api/ store/ ui/ hooks/`).
- `node_modules` корректно не закоммичен (и в корневом, и во frontend `.gitignore` есть правило).

## 3. Проблемы — Critical (чинить в первую очередь)

1. **`DeleteUserHandlerService` проверяет не того пользователя — авторизация полностью
   инвертирована.**
   `internal/usecase/user/deleteUser.go:10-13` + `internal/controller/http/user/usersHandlers.go:341,359`.
   Хендлер читает `login` вызывающего из контекста, но **никогда не передаёт его** в
   `DeleteUserHandlerService(idInt)`. Внутри сервис берёт **удаляемого** пользователя по `id`,
   затем зовёт `checkUserRole(user.Login, "Owner")` — то есть проверяет роль **цели**, а не
   того, кто удаляет. Итог: любой авторизованный пользователь может удалить любого не-Owner'а,
   а Owner'а не может удалить вообще никто (в т.ч. другой Owner). Проверено чтением кода напрямую.

2. **Управление классами полностью без проверки прав.**
   `internal/controller/http/class/classesHandlers.go`: `AddClassHandler` (423),
   `UpdateClassHandler` (113), `AddParallelClassHandler` (47), `DeleteClassHandler` (475),
   `DeleteParallelClassHandler` (182), `CompleteQuarterHandler` (305), `YearComplete` (336) — ни
   один не проверяет роль/права. Только `SetClassTeacherHandler` (642) зовёт
   `CanManageClasses`. Проверено: `ClassUsecase` (`internal/usecase/class/usecase.go`) вообще не
   хранит ссылку на permission-сервис или репозиторий пользователей — проверять права там просто
   нечем. Любой залогиненный пользователь (в т.ч. ученик) может создавать/удалять классы,
   завершать четверть, переводить школу на новый год.

3. **`updateNetscapeCookiesFile` пишет живые access/refresh токены в открытом виде на диск при
   каждом логине/рефреше.**
   `internal/controller/http/auth/auth.go:157-216`, вызывается из `LoginHandler` (69) и
   `RefreshHandler` (123). Файл `cookies.txt` в формате Netscape cookie-jar содержит рабочие
   токены сессии без какого-либо срока жизни/ротации/флага окружения. Это классический вектор
   утечки токена (бэкап, случайный коммит, чтение с диска другим процессом). Судя по всему,
   остаток отладки через `curl -b cookies.txt`. **Нужно удалить полностью до продакшена.**

4. **Миграция схемы `ensureColumn` — no-op, никогда не добавляет колонку.**
   `internal/adapter/postgres/storage/schema.go:554-571`. Функция проверяет
   `information_schema.columns`, и если колонки нет — просто `return err`, где `err == nil`
   (после успешного `Scan`). `ALTER TABLE ... ADD COLUMN` нигде не вызывается. Используется для
   ~15 колонок (`schedules.WeekType`, `users.ClassID`, `events.RatingReward`,
   `event_params.ExtraRatingReward`, `parallels.BestClassID` и т.д.) — на существующей БД без этих
   колонок приложение молча продолжит стартовать, а затем упадёт в рантайме на первом же запросе,
   ссылающемся на отсутствующую колонку, вместо явной ошибки при старте. Проверено чтением кода.

## 4. Проблемы — High

5. **Cookie logout не совпадает по `path` с cookie логина — logout не удаляет реальную cookie
   браузера.** `auth.go` (`setAccessTokenCookie`/`setRefreshTokenCookie`) ставит cookie на путь
   `"/api"`; `usersHandlers.go` `LogoutHandler` очищает cookie на пути `"/backend/api"`. Браузер
   различает cookie по path, поэтому очистка создаёт отдельную пустую cookie на другом пути, а
   реальная сессионная cookie на `/api` остаётся в браузере рабочей. Blacklist из Redis частично
   компенсирует (сервер отклонит токен), но пока Redis жив — единственная защита; если Redis
   очистить/уронить, старая cookie снова станет валидной.

6. **`UpdateAvatarHandler` — пропущен `return` после ошибки `Atoi`, плюс IDOR.**
   `internal/controller/http/user/usersHandlers.go:96-99`. При ошибке парсинга `id` пишется
   500, но выполнение продолжается с `id == 0`, и хендлер всё равно вызывает
   `userService.UpdateAvatar(in, id)` — двойной ответ + аватар нулевого пользователя. Отдельно:
   `id` берётся из query-параметра без проверки, что это аватар **своего** аккаунта — любой
   пользователь может перезаписать чужой аватар, зная его ID. Проверено чтением кода.

7. **Rating usecase — гонка чтение-изменение-запись, "мёртвый" метод инкремента.**
   `internal/usecase/rating/usecase.go`. `RatingsUsecase` хранит `repo.RatingRepository` (со
   единственным методом `UpdateRating`, судя по всему — атомарный `SET rating = rating + $1`),
   но реальный код `UpdateRating` (в usecase) его не вызывает вообще: читает пользователя,
   меняет `Rating` в памяти, сохраняет через `SaveUser` (полная перезапись строки). Два
   одновременных изменения рейтинга одного пользователя потеряют одно из изменений
   (classic lost update). Атомарный путь в интерфейсе существует, но не используется.

8. **`internal/usecase/user/deleteUser.go:15` — сравнение ошибок через `==`, всегда `false`.**
   `if err == errors.New("user not found")` — два независимо созданных `*errors.errorString`
   никогда не равны в Go. Ветка мертва, "not found" никогда не обрабатывается отдельно.

9. **`POST /api/refresh` не защищён rate limiting.** Rate limiting подключён только в
   `AuthUsecase.Login` (`internal/usecase/auth/ratelimit.go`), то есть только на `/login`.
   `/api/refresh` (`router.go:57`, публичный роут вне `AuthMiddleware`) принимает refresh-токен
   из cookie без какого-либо троттлинга — не критично само по себе (токен 32 случайных байта),
   но асимметрично по сравнению с login и не логируется/не алертится отдельно.

10. **Открытый API-ключ конфигурации: `openPostgres` игнорирует `cfg.DBPath`.**
    `internal/adapter/postgres/storage/postgres.go:11` — `func openPostgres(_ string)` читает
    `os.Getenv("DB_PATH")` напрямую, параметр отброшен. `config.Config.DBPath`
    (`internal/adapter/config/config.go`, дефолт `"data/storage.db"` — похоже на путь для SQLite,
    хотя используется Postgres) загружается, дефолтится и передаётся в `storage.New(cfg.DBPath)`,
    но нигде не используется по факту. Мёртвый и вводящий в заблуждение путь конфигурации.
    Проверено чтением кода.

11. **`main.go` не вызывает `os.Exit(1)` при фатальных ошибках старта.** Ошибка конфига,
    открытия БД, `MkdirAll`, инициализации параллелей/seed, даже падение `r.Run()` — везде
    просто `slog.Error(...); return` из `main()`. Процесс завершается с кодом `0`, как будто всё
    прошло успешно. Docker/systemd/Kubernetes health-check по exit-коду не увидят падения при
    старте и не перезапустят/не заалертят.

12. **Многошаговые записи user↔class не в транзакции.** `AddUser`, `SaveUser`, `DeleteUser`
    (`internal/adapter/postgres/user/userPostgresRepo.go`) пишут в `users`, затем отдельным,
    необёрнутым вызовом зовут `syncClassByIDLocked` (пересчёт `Members`/`UserTotalRating`/
    `TeacherLogin` в `classes`). При падении между шагами агрегаты класса останутся рассогласованы
    с реальным составом без отката.

## 5. Проблемы — Medium

13. **Инконсистентная валидация между структурно одинаковыми сущностями.**
    `NoteUsecase.AddNewNote` валидирует `TargetID`/`Content` и сам заполняет `AuthorID`; у
    `ComplaintUsecase.AddNewComplaint` (`internal/usecase/complaint/usecase.go:62-82`) валидации
    нет вообще, и в структуру напрямую пробрасывается **клиентский** `in.ID` — притом что Note
    сознательно оставляет `ID` нулевым для БД.
14. **Проглоченные ошибки при удалении.** `DeleteComplaint`/`DeleteNote` логируют фиксированную
    строку без `err.Error()` (в отличие от `event`/`schedule`, где сообщение дописывается),
    возвращают клиенту generic `"server error"` — реальная причина падения теряется полностью.
15. **`UpdateUserHandlerService` обходит инвариант рейтинга.** Клиент может прислать
    произвольный `Rating` через `UpdateUserRequest` без клэмпа `[0, 5000]`, который в
    `rating/usecase.go` соблюдается только на одном из двух путей записи в ту же таблицу
    `users`. Схема БД (`Role TEXT NOT NULL`) тоже не имеет `CHECK`-ограничения на допустимые роли.
16. **Дублирующаяся, разная по строгости проверка роли.** `deleteUser.go:checkUserRole`
    (case-sensitive) и `permission.go:CheckUserRole` (case-insensitive, `strings.ToLower`) — два
    разных определения "имеет ли пользователь роль X" в одной кодовой базе.
17. **Домен-сущности используют `database/sql.NullString` напрямую.**
    `internal/domain/user/entity.go` — `User.Avatar`, `SafeUser.Avatar`,
    `UpdateUserRequest.Avatar *sql.NullString` — инфраструктурный тип протекает в домен и в JSON,
    отдаваемый клиенту.
18. **Широкая утечка внутренних сообщений об ошибках клиенту.** `err.Error()` передаётся прямо в
    JSON-ответ во множестве мест (`usersHandlers.go:225,311,360`, `classesHandlers.go:87,93,142-144`,
    `notesHandlers.go`, `complaintsHandlers.go`, `ratingsHandlers.go`, `scheduleHandlers.go`) —
    может раскрыть детали БД/внутренних полей атакующему. Часть GET-хендлеров делает это правильно
    (generic-сообщение), часть — нет.
19. **Несогласованные статус-коды для одной и той же ситуации.** Провал проверки роли в
    `event`-хендлерах (`AddEventHandler`, `AddEventParams`, `UpdateEventHandler`, `EventComplete`
    и др.) отдаёт `500` вместо `403`, тогда как note/complaint/schedule для того же случая
    (через `errors.Is` на `ErrAccessDenied`) корректно отдают `403`.
20. **CORS не настроен вообще** (ни одного упоминания `cors`/`Access-Control-*` в кодовой базе).
    Пока не проблема безопасности (браузер просто блокирует кросс-доменные запросы), но при
    первом же разнесении фронта и бэка по разным origin потребует аккуратной настройки — с
    cookie-based авторизацией сочетание `AllowAllOrigins + AllowCredentials` было бы уже реальной
    дырой, так что важно не срезать угол при добавлении.
21. **`Secure`-флаг cookie по умолчанию выключен.** `cookieSecure()` возвращает `false`, если не
    выставлена `COOKIE_SECURE=1` явно — легко забыть на проде, тогда токены пойдут через HTTP
    в открытом виде.
22. **72-часовой access-токен без ротации — при этом единственный механизм отзыва (blacklist)
    fail-open.** При недоступности Redis отозванный логаутом токен **снова считается валидным**
    — это менее очевидный компромисс, чем fail-open для rate limiting логина (там всего лишь
    временно снимается защита от подбора, а не откатывается явное действие пользователя).
    Работает "как задумано", но стоит завести алерт поверх лога `"redis unavailable, failing
    open"`, а не оставлять его неотличимым от обычного info-лога.
23. **`Increment` в Redis-адаptere не атомарен целиком.** `internal/adapter/redis/cache/cacheRepo.go`
    — если `INCR` создал ключ (`count == 1`), но следующий `EXPIRE` упал по сети, ключ остаётся
    без TTL навсегда. Редкий, но реальный краевой случай (стоит обернуть в `TxPipelined` или
    Lua-скрипт, если критично).
24. **`internal/usecase/auth/logout.go` использует `ParseUnverified`.** Безопасно в текущем
    единственном месте вызова (роут `logout` уже стоит за `AuthMiddleware`, который проверяет
    подпись раньше), но метод публичный и ничем не защищён от вызова с непроверенной подписью в
    будущем — переиспользование в другом месте могло бы дать возможность форс-подсунуть большой
    `exp` и раздуть TTL в Redis. Дешёвая защита — проверять подпись и здесь тоже, раз `jwtSecret`
    уже есть в `AuthUsecase`.
25. **`backend.zip` (~158 КБ) закоммичен в корне репозитория** и продолжает изменяться между
    сессиями — постороннее избыточное вложение, раздувающее историю git. Похоже, что какой-то
    внешний процесс синхронизирует эту зип-копию `backend/` — в ходе этой самой сессии он привёл
    к тому, что только что созданный `backend/internal/adapter/redis/README.md` исчез с диска и
    его пришлось пересоздавать. Стоит выяснить источник этого файла и либо убрать его из git,
    либо (если это осознанный бэкап-механизм) исключить редактируемые директории из его области
    действия, чтобы он не расходился с рабочим деревом.
26. **`frontend/.env.development` закоммичен в git.** Содержимое сейчас безобидное
    (`VITE_API_URL=` пусто), но сам факт трекинга `.env.*`-файла — плохая привычка: стоит
    переименовать в `.env.development.example` или явно добавить в `.gitignore`, пока туда не
    попал настоящий секрет.

## 7. Проверено и опровергнуто (false positive)

Один из агентов заявил, что `UpdateRatingsHandler` (`internal/controller/http/rating/ratingsHandlers.go`)
не имеет проверки прав и позволяет любому пользователю менять чужой рейтинг. **Это неверно** —
проверено чтением кода: хендлер прокидывает `needUser` в `RatingsUsecase.UpdateRating`
(`internal/usecase/rating/usecase.go`), которая **внутри** явно проверяет
`permissionService.CanManageClasses(user.Role)` и возвращает ошибку `"only admins and owners can
update ratings"` для всех остальных ролей. Проверка прав здесь корректно живёт в usecase-слое
(как и должно быть в Clean Architecture), а не в контроллере — это не баг, а другой (правильный)
слой реализации, чем ожидал агент. Оставляю эту заметку в отчёте намеренно — чтобы отчёт был
проверяемым, а не просто пересказом того, что сказали суб-агенты.

## 9. Итоговая оценка

### Уровень проекта

Проект крепкого **middle-уровня по архитектуре** и **junior/middle-уровня по операционной
дисциплине безопасности**. Разрыв между этими двумя половинами — главная характеристика
кодовой базы:

- Инфраструктурный скелет (Clean Architecture, параметризованный SQL, транзакции там, где до
  них дошли руки, разумный pool БД, bcrypt, защита от JWT `alg`-confusion, грамотная деградация
  Redis) — сделан на уровне, который был бы уместен в куда более крупном проекте. Это не
  "студенческий" код в смысле структуры.
- При этом авторизация — самое слабое место: три независимых, серьёзных дыры (удаление
  пользователя, управление классами, конкретно точечно проверенный рейтинг оказался в порядке —
  но статистически это означает, что "проверять права" не стало рефлексом, а не то, что все
  места защищены). Это типичная ошибка на стыке "быстро добавили фичу" и "забыли, что controller
  и usecase — это про контракт, а не только про данные".
- Обработка ошибок неровная: где-то (schedule, новый auth/redis код) — образцовая, где-то —
  проглатывается или сравнивается указателями по `==`.
- Тестами покрыто ~20% функциональности (3 файла на 9 доменов), и именно auth/permission —
  самое чувствительное — не покрыто вообще.

Итоговая оценка проекта: **работоспособный pet/учебный проект с реальной архитектурной
культурой, который прямо сейчас нельзя показывать в интернет без исправления критических пунктов
из чек-листа выше** — не потому что архитектура плохая, а потому что несколько конкретных
мутирующих эндпоинтов буквально не проверяют, кто их вызывает.

### Ты как разработчик

Ты явно вышел за рамки типичного "просто заставить работать": самостоятельно выстроил
многослойную архитектуру и выдержал её единообразно на 8+ сущностях, без единого места, где
кто-то "срезал угол" через прямой импорт adapter'а в usecase. Умение читать существующий код
и следовать заданному паттерну (что ты и продемонстрировал в работе над Redis-слоем в этой же
сессии — порт/адаптер, fail-open, хэширование токена, доккомментарии) — это mid-level навык,
а не junior.

Слабое место — не архитектура, а **дисциплина проверки самого себя**: часть критических находок
(инвертированная проверка при удалении пользователя, полностью незащищённые class-эндпоинты,
`err == errors.New(...)`) — это не "не знал, как правильно", а "написал и не перечитал/не
протестировал сценарий не-своего пользователя". Это ровно то, что ловится либо тестами на permission-слой,
либо привычкой на каждый мутирующий хендлер спрашивать "а что если это вызовет чужой аккаунт/не тот
пользователь?" — привычки пока не видно как системной (там, где она есть — schedule, note/complaint
GET — сделано образцово; там, где её нет — class, delete user — дыра).

Итог: **уверенный middle по архитектурному мышлению, junior/middle по security-рефлексам и
тестовой дисциплине.** Ближайший наибольший рычаг роста — не изучать новые паттерны, а взять
за привычку после каждой мутирующей ручки писать (себе, не обязательно как формальный тест)
сценарий "что если вызывающий — не тот, кем должен быть", и заводить хотя бы по одному
happy-path + одному permission-path тесту на новый usecase, а не откладывать тесты на "потом".
